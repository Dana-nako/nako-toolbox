# なこ用ツール箱 設計メモ v1.1

## 1. 目的

「なこ用ツール箱」は、なこが日常的に使う小さな補助ツールをまとめるためのWeb/PWAツール群である。

最初の中心は創作依頼補助ツールだが、将来的には生活補助・記録補助・実験ツールも同じ入口から扱えるようにする。

ただし、入口はひとつでも、中身はジャンルごとに分けて管理する。

## 2. 基本方針

- スマホ優先で設計する。
- PCでも使えるようにし、画面幅に応じてレイアウトを切り替える。
- 公開リポジトリには、公開されても困らない空の道具本体だけを置く。
- 個人情報、内側の名前、変換ルール、生活ログ、非公開メモはGitHubに置かない。
- 設定データは端末内保存を基本とし、手動バックアップ/復元を用意する。
- 将来的に同期や外部保存へ移行できるよう、データ構造は整理しておく。

## 3. ツール分類

トップページは「入口」として使い、下位にジャンル別の棚を持つ。

想定ジャンル：

- 創作補助
  - 名前置換メーカー
  - ワード変換メーカー
  - プロンプト整形系ツール
- 生活補助
  - 冷蔵庫メモ
  - 買い物メモ
  - 献立補助
- 記録・管理
  - 設定バックアップ
  - ツール箱全体バックアップ
- 実験室
  - 分類未確定の試作品

新しい要望が出た場合、すぐ作らず、まず以下を判定する。

1. 新ツールにするか
2. 既存ツールの拡張にするか
3. Notionなど外部ツール向きか
4. Web/PWAツール向きか
5. 今作るべきか、後回しにすべきか
6. 公開リポジトリに置いて安全か
7. 同期が必要か

## 4. 公開安全ルール

GitHub PagesはPublic運用になる可能性があるため、公開安全を前提にする。

GitHubに置いてよいもの：

- HTML/CSS/JavaScript
- 空のUI
- 公開されても困らない説明文
- アイコンや背景など公開可能な素材
- 公開安全な設計メモ

GitHubに置かないもの：

- 個人情報
- 非公開の内側名
- 変換ルールの実データ
- 生活ログ
- 食事・体重などの記録
- 非公開の相談ログ
- なこの本音や内部判断が含まれるメモ

## 6. 名前置換メーカーのデータ方針

名前置換メーカーは、原則として **1キャラ1名前** で扱う。

保存形式：

```json
{
  "id": "character-id",
  "name": "表示名"
}
```

運用ルール：

- キャラ名はHTML/JavaScriptへ初期値として直書きしない。
- キャラ名は端末内のlocalStorageへ保存する。
- GitHubへ置くコードは、キャラ一覧が空でも動く状態にする。
- 新規追加は、ユーザーが入力した名前をそのまま `{NAME}` に入れる。
- プルダウン式の複数呼び名機能は採用しない。
- 旧形式の `variants` / `selectedVariant` は、旧データを単一名へ移行する時だけ読む。
- 旧データを読み込んだ場合は、`selectedVariant`、`baseLabel`、`variants[0]` の順で単一名へ変換し、現行形式で保存し直す。

## 6. データ保存方針

初期方針は、端末内保存と手動バックアップで運用する。

- 各ツールの設定は localStorage または IndexedDB に保存する。
- テキスト中心の設定は localStorage を使う。
- 画像など大きいデータを扱う場合は IndexedDB を検討する。
- ブラウザデータ削除で設定が消える可能性があるため、バックアップ/復元機能を必須とする。
- スマホとPCの自動同期は初期実装では行わない。
- 将来の同期に備え、全体バックアップ形式を設計しておく。

将来的な全体バックアップのイメージ：

```json
{
  "version": 1,
  "tools": {
    "nameReplacer": {
      "characters": []
    },
    "wordReplacer": {
      "rules": []
    }
  },
  "ui": {
    "theme": "black-gold"
  }
}
```

## 7. デザイン管理方針

見た目は各HTMLに直書きせず、共通CSSで管理する。

想定構成：

```text
assets/
  css/
    app.css
  js/
    common.js
    dashboard.js
    name-replacer.js
    word-replacer.js
  images/
    dana-dashboard.jpg
icons/
  icon-192.png
  icon-512.png
```

役割：

- app.css
  - 全ページ共通の土台
  - 色、余白、カード、ボタン、入力欄、レスポンシブ調整
- common.js
  - コピー処理、通知、保存補助など
- dashboard.js
  - Service Worker登録など、ダッシュボード固有処理
- name-replacer.js
  - 名前置換メーカー固有処理
- word-replacer.js
  - ワード変換メーカー固有処理

CSS変数を使い、色や角丸を一元管理する。

例：

```css
:root {
  --bg: #101114;
  --panel: #181a20;
  --text: #f2f0e8;
  --accent: #f2c14e;
  --accent-strong: #d4af37;
  --radius: 18px;
}
```

## 8. テーマ方針

初期テーマは少数に絞る。

初期候補：

- black-gold
  - 標準テーマ
  - 黒地＋金アクセント
- soft-blue
  - 生活補助向け
  - 明るめ・落ち着いたテーマ
- lab-orange
  - 実験室向け
  - 試作・遊び用テーマ

全ページの骨格は共通にし、ジャンルごとにアクセント色や背景だけを変える。

やりすぎて各ページが別アプリのように見える状態は避ける。

## 9. 背景画像方針

初期段階では、背景画像はダッシュボードだけに使用する。

- ダッシュボード背景は、なこが選んだダナの絵を1枚固定で使う。
- ランダム背景は初期実装では行わない。
- 時間帯/曜日による背景変更も初期実装では行わない。
- 背景設定機能も初期実装では行わない。
- ツール画面は読みやすさ優先とし、背景演出は控えめにする。

背景画像を使う場合は、必ず暗めのオーバーレイを重ねて文字の可読性を守る。

スマホとPCでは見え方が異なるため、必要に応じて表示方法を分ける。

- スマホ
  - 縦長背景を活かす
  - 入口としての雰囲気を出す
- PC
  - 背景を弱める
  - 無理に拡大しない
  - 必要なら背景なしに近い見せ方にする

## 11. スマホ/PCレイアウト方針

スマホ優先で設計し、PCでは横幅を活かす。

スマホ：

- 縦積み
- ボタン大きめ
- 入力欄広め
- 片手でも押しやすい
- コピー/生成など主要操作を分かりやすくする

PC：

- 入力と出力を横並びにできる場合は検討する
- 一覧性を上げる
- 余白を活かす
- スマホ用レイアウトを無理に引き伸ばさない

## 12. 共通機能方針

共通化する機能：

- コピー処理
- コピー失敗時の手動コピー欄
- localStorage保存/読み込み補助
- バックアップ/復元処理
- ステータス表示
- エラー表示
- ツール箱へ戻るリンク
- テーマ適用

各ツール固有の処理だけを各HTMLまたは専用JSに残す。

## 13. 追加ツールの判定フロー

新しい要望が出た時は、次の順で確認する。

1. 何をしたいのか
2. いつ使うのか
3. スマホで使うのか、PCで使うのか
4. 入力するデータは何か
5. 出力として何が欲しいのか
6. 保存は必要か
7. 同期は必要か
8. 個人情報や非公開情報を扱うか
9. 既存ツールで代用できるか
10. 試作品として作るか、本番ツールとして作るか

「欲しい」と言われたものをそのまま作らず、用途と保存データを確認してから設計する。

## 14. 開発・確認フロー

今後の基本フロー：

```text
相談
→ 要件メモ
→ 確認用プレビュー
→ なこがスマホで挙動確認
→ OKならGitHub用ZIP
→ GitHubへ反映
→ スマホで本番確認
```

本番GitHubへ直接反映する前に、必ず確認用プレビューで動作を見る。

確認項目：

- スマホで表示が崩れないか
- ボタンが押しやすいか
- 追加/削除/保存が動くか
- バックアップ/復元が動くか
- コピーが動くか
- 文字が読みにくくないか
- 古いキャッシュを掴んでいないか

## 15. 現在の決定事項

- ツール箱の入口はひとつ。
- 中身はジャンルごとに分ける。
- スマホ優先。
- PCでも画面幅に合ったレイアウトにする。
- GitHubには公開安全な空の道具本体だけを置く。
- 内側のデータはGitHubに置かない。
- 設定は端末内保存＋手動バックアップから始める。
- 将来の全体バックアップを見越す。
- 共通CSS/共通JSへ構造整理する。
- ダッシュボードだけ背景画像を固定1枚で試す。
- 背景画像のランダム化・時間帯切り替え・ユーザー追加機能は後回し。
- 本番反映前に必ずプレビュー確認を行う。

## 16. 次にやること

1. この設計メモを確認する。
2. 問題なければ、現在のツール箱を共通CSS/共通JS構成へ作り直す。
3. 確認用プレビューを作る。
4. 名前置換メーカーとワード変換メーカーの挙動を確認する。
5. ダッシュボード背景用の画像を実装直前に受け取る。
6. ダッシュボード背景を組み込む。
7. プレビューで確認する。
8. OKならGitHub用ZIPを作る。

## 現行運用メモ

- 本番へ上げる前に、単一HTMLの確認だけで終わらせない。
- GitHub用へ分割した後のHTML/CSS/JavaScript構成でも、ボタン操作を再確認する。
- JavaScriptを分割した場合は、相対パスを実ファイル基準で確認する。
- Service Workerを変更した時は、`CACHE_NAME` を必ず更新する。
- GitHub Pages反映後は、ブラウザ表示とPWA表示の両方を確認する。
- 書き込み権限がないため、現状の更新はZIP展開→GitHubへ上書きで行う。
